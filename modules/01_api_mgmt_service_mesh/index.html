API Management Concepts

 

APIs Management fosters relationships between API Consumers and API Providers

API Management to date has been less focused on Infrastructure and Traffic Management and more focused on Developers, Applications and Business aspects of an API

 

Core concepts include

  • Developers, Applications, Application ID/Key, API, Methods, Metrics, Sign-ups and Subscriptions, Usage Limits, Pricing Rules, Plans, Billing, Developer portal, API Consumer, API Provider, API Documentation, Active Docs,

 

Analytics about usage maybe highlight a top developer, or an application trending up, or the spread of usage across an APIs methods - giving insight into how an API is used and by whom.

 

So, the difference between API Management and Traffic Management, is not so much where the traffic is flowing, but what concepts are used to view & control the traffic.

API Management Feature set

API Provider:

  • Create and publish a Developer Portal with API documentation, customized for each API Consumer

  • Packaging of APIs into Plans to be offered to API Consumer on the Developer Portal. Plans may contain:

    • Limits the set of methods of the API that can be used in a given plan.

    • Rate Limit the usage of API methods (or metrics derived from usage), by defining different limits over different periods of time, in each plan.

    • Cost calculation based on API usage via pricing rules (fixed, variable or time driven)

  • API Presentation: Presenting a “clean, uniform” API externally where requests are then routed to multiple internal services by the API Gateway based on “mapping rules”

  • Billing to API Consumers based on their usage

  • Authorization: Confirm that the API request coming from an app is in fact authorized to proceed, collaborating in OAuth/OIDC flows in conjunction with IDP providers

  • Analytics to understand the usage of the API by API Consumers

  • Admin capabilities for apps, developers, keys, APIs etc.

API Management Feature set

API Consumer:

 

  • API Documentation (interactive and testable if possible) on a Developer Portal

  • A mechanism for subscription to a plan (ie: obtain keys) & other info related to the Consumer’s subs

 

Shift in API usage and API Management

Evolution of API Management

  • Until recently, the main focus of API Management at 3scale has been to manage REST APIs that organizations are opening up to outside developers/partners/customers who are not part of the same organization

  • API providers define APIs, then package them in plans with different capabilities, permitted usage levels and possibly prices. Usage of an API is then offered to “API consumers” via subscription to one of these plans

  • API consumers require documentation on APIs, to understand the methods available to them, to be able to try them, sign-up to the API (possibly paying for it), create applications that can use the API and get API keys that applications use to access the API

The Shift

There has been an increase in the use of REST APIs between systems, applications, components or services inside an organization via what 3scale calls “Internal APIs”.

As the number of services increases this becomes more a MicroServices Architecture (MSA) discussion.

  • Whether the invocation of one service from another via a method such as http is considered an “API call” in the traditional sense or not, is part of the shift being discussed.

  • In these cases the “API consumers” (developers of the service invoking another one) will often belong to the same organization, and the usage will seldom be for pay.

  • The “application” consuming the other service’s API is actually an internal service.

  • The provider of the service may still wish to limit usage, track usage via analytics and potentially put an internal cost on the usage through pricing rules (whether actually charged or not).

  • In larger organizations with many developers and many services, there is still a need for Service Discovery (by humans), and to understand them via documentation and to have API documentation that also allows easy trial use of them.

Features Requested of API Management

Advanced Traffic Management

3scale gets requests for features related to Traffic management, routing and manipulation for the traffic going through the API Gateway. Examples are:

  • Mapping public paths to multiple internal backends (implemented by AMP)

  • IP Blacklisting / Whitelisting (mainly for Ingress)

  • URL Mapping and rewriting

  • CORS

  • Header adding

  • Content based routing

  • Transactions per second limits (by application or end user)

  • Connection number limiting

  • Throttling

  • Tracing

  • Circuit breaker

 

Some of these are related to API Management and some are less so. Including some or all of these features into the API Management solution of Red Hat increases it’s surface area significantly, and take it’s COG away from the Business/Developer/Provider/Docs areas and more into Traffic and Infrastructure management.

MSA / Service Mesh / Istio Feature Set

 

Istio and Service mesh management is mainly about assigning identity, for the benefit of traffic control into and within the cluster/mesh:

  • Service discovery

  • Automatic retries (hides failures from clients)

  • Timeouts

  • Load balancing

  • Rating and limiting

  • Circuit breaker

  • Bulk heading

  • Authorization and authentication

  • Routing (DMZ, Surgical, Dynamic for A/B and Canary, Mirroring, Shadow traffic)

  • Fault injection

  • App level and end-to-end monitoring

  • Logging

  • Metrics

  • Distributed tracing
